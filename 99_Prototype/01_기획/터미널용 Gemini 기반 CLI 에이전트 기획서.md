# 터미널용 Gemini 기반 CLI 에이전트 기획서

## 1. 배경 및 문제 인식

구글의 Gemini 모델은 웹, IDE, API 등 여러 채널에서 잘 쓰이고 있지만, 개발자 입장에서는 "내가 이미 열어둔 터미널 안에서 모든 걸 해결하고 싶다"는 요구가 강하다. 특히 초보~중급 개발자는 명령어를 복사·붙여넣기만 하고, 해당 명령이 무엇을 의미하는지, 에러 메시지가 왜 나오는지, 지금 상황이 정상인지 비정상인지 스스로 판단하기 어렵다.[^1][^2]

또한 일부 에이전트/툴은 매번 명령 실행 전에 "accept?"와 같은 확인을 요구하는 UX를 가지는데, 사용자는 이게 정확히 무엇을 승인하는지 모른 채 습관적으로 승인 버튼을 누르게 되며, 이는 학습에도, 보안에도 좋지 않다. 인텔 맥처럼 환경 제약이 있는 기기에서 Node.js 설치 실패처럼 반복적인 시행착오를 경험할 때, 그 원인을 함께 분석해주고 다음 단계까지 안내해 줄 수 있는 CLI 동반자가 필요하다.[^2]


## 2. 기존 솔루션 리서치 요약

### 2.1 Gemini CLI (공식)

Google은 이미 "Gemini CLI"라는 오픈소스 AI 에이전트를 공개하여, 터미널에서 직접 Gemini 모델을 사용할 수 있게 하고 있다. Gemini CLI는 ReAct(reason + act) 루프를 사용해 로컬 터미널, 파일 시스템, 웹 검색 등의 툴을 호출하여 버그 수정, 기능 개발, 테스트 개선 등 복잡한 개발 작업을 수행할 수 있다.[^3][^4]

공식 블로그에 따르면 Gemini CLI는 코드 이해·생성, 파일 및 디렉터리 조작, 명령 실행과 동적 트러블슈팅 등 터미널 경험 전반을 업그레이드하는 것을 목표로 하며, 개인 Google 계정만으로도 넉넉한 무료 사용량을 제공한다. 한국어 문서에서도 `/memory`, `/stats`, `/tools`, `/mcp`, `grep`, `terminal`, `file read/write`, 웹 검색 등 다양한 내장 명령과 도구 사용을 안내하고 있다.[^4][^1]

### 2.2 한국어 사용 후기/가이드

국내 블로그와 기술 블로그에서는 Gemini CLI 설치 방법, 기본 사용법, 명령어 목록, 실전 예제를 상세히 다루고 있다. 예를 들어 `/help`, `/auth`, `/memory`, `/config`, `/history`, `/retry`, `/mode`, `/log`, `/workspace` 등 명령을 이용해 인증, 메모리 관리, 설정, 작업 디렉터리, 도구 사용 모드(자동/수동) 등을 제어하는 예시가 소개된다.[^5]

또 다른 한국어 가이드에서는 Gemini CLI를 "터미널 환경에서 직접 작동하는 대화형 AI 코딩 어시스턴트"로 설명하면서, 파일 시스템 조작, 셸 명령 실행, Git 워크플로 자동화, 에러 해결 등 개발자가 자주 하는 작업을 자연어로 위임하는 흐름을 보여준다.[^2]

### 2.3 유사한 셸 AI 도구들

Gemini 전용은 아니지만, 자연어를 셸 명령으로 변환하고 설명해주는 오픈소스 도구들도 존재한다. 예를 들어 ShellAI는 "터미널 어시스턴트"로, 자연어 설명을 기반으로 셸 명령을 생성하고, 짧은 설명과 함께 결과를 보여주는 CLI 툴이다. AI Shell 역시 자연어를 기반으로 안전한 셸 명령을 생성하고 실행 전 설명을 제공하며, 대화형 모드로 여러 단계의 명령을 설계할 수 있게 한다.[^6][^7]

또 다른 예로 AskCLI는 로컬 모델 또는 상용 API(OpenAI, Google Gemini 등)를 백엔드로 사용하여, 리눅스/유닉스 셸에서 간단한 AI 어시스턴트를 제공하는 실험적 도구이다. 이들 도구는 공통적으로 "설명과 명령어 생성"에 강점이 있지만, 사용자의 학습 여정 전체를 설계하고, 각 명령과 출력에 대해 단계별 피드백을 주는 "튜터형" UX는 상대적으로 약하다.[^8]


## 3. 페인 포인트 정리

### 3.1 현재 경험의 문제점

- **반복되는 승인(accept) UX**  
  - 일부 에이전트는 명령 실행마다 승인 절차를 요구하지만, 사용자는 무엇을 승인하는지 구체적으로 알지 못한다.[^2]
  - 이는 보안 상으로는 안전해 보이지만, 실제로는 "생각 없이 y를 누르는 습관"만 만들어 학습 효과가 떨어진다.

- **복사·붙여넣기 위주의 사용**  
  - 웹에서 찾은 명령어를 그대로 복사해 터미널에 붙여넣을 뿐, 명령의 의미, 옵션, 부작용을 이해하지 못한다.[^7]
  - 문제 발생 시 원인을 스스로 추적하기 어려워, 다시 검색하거나 질문을 반복해야 한다.

- **에러 메시지 이해 부족**  
  - Node.js 미지원 아키텍처(예: 일부 인텔 맥)와 관련된 오류처럼 환경 의존적인 문제는 에러 메시지만으로 직관적으로 이해하기 어렵다.[^2]
  - 어떤 부분이 환경 문제이고, 어떤 부분이 설정 실수인지 구분해주는 설명이 필요하다.

- **컨텍스트 전환 비용**  
  - 브라우저 기반 IDE/에이전트(예: 웹 UI, Anti-gravity 류 도구)를 사용할 때, 매번 브라우저로 전환하고 승인 버튼을 누르는 과정이 귀찮고 흐름이 끊긴다.[^1]
  - 터미널만 띄워놓고, 질문·명령·확인·학습이 한 자리에서 이뤄지길 원한다.

### 3.2 사용자의 목표/성향

- 실무 엔지니어이지만, Node.js/프론트엔드/새로운 도구들에는 "실험적인 초보"라고 스스로 인식하고 있음.  
- 단순히 문제를 해결하는 것을 넘어서, "내가 지금 무슨 명령을 치고 있고, 이 결과가 무엇을 의미하는지"를 이해하며 배우고 싶어 함.[^5]
- 최대한 자동화하되, 중요한 시점에서는 왜 그런 행동을 하는지 설명을 듣고 싶은 "반자동 + 튜터" 스타일의 경험을 선호.


## 4. 목표 시스템 개요

이 기획서에서 상정하는 시스템은 다음과 같다.

- **형태**: 터미널 내부에서 작동하는 CLI 에이전트(별도 명령어, 예: `gai`, `g tutor` 등)  
- **모델**: Google Gemini (공식 Gemini CLI를 그대로 활용하거나, 동일 API를 사용하는 래퍼)  
- **역할**: 
  - 셸 명령어 설계/추천
  - 명령 실행 전 후 설명 및 영향 분석
  - 에러/로그 해석 및 다음 단계 제안
  - 초보 개발자를 위한 학습형 튜토리얼 플로 제공

Gemini CLI 자체가 이미 강력한 기반을 제공하므로, "완전히 새로운 것을 만드는 것"보다는 Gemini CLI를 기본 엔진으로 사용하고, 그 위에 학습/튜터 기능을 얇게 얹는 아키텍처가 현실적이다.[^3]


## 5. 업무 정의(에이전트가 해줘야 할 일)

### 5.1 CLI 코파일럿 역할

- 자연어 요구 사항을 기반으로, 현재 OS/아키텍처에 맞는 셸 명령어 세트를 설계한다.[^4]
- 각 명령어에 대해 "무엇을 하는지" 한 줄 요약과, 옵션에 대한 간단한 설명을 제공한다.[^6]
- 위험도가 있는 명령어(`rm -rf`, `sudo`, 시스템 설정 변경 등)는 별도의 경고와 대체 옵션을 제시한다.[^7]

### 5.2 실행 관리자 역할

- 사용자가 선택한 명령을 실제 셸에서 실행하고, 표준 출력/에러를 캡처한다.[^3]
- 종료 코드(0/비0)에 따라 성공/실패 여부를 판단하고, 실패 시 원인 후보와 다음 시도 옵션을 제안한다.[^2]
- 반복 실패 시 동일 패턴을 감지하고, 보다 근본적인 원인(예: "이 맥OS 버전에서는 공식 Node 패키지가 인텔을 더 이상 지원하지 않음")을 추론하도록 유도한다.[^2]

### 5.3 학습/튜터 역할

- 사용자가 "이 명령이 무슨 의미인지 설명해줘"라고 요청하지 않아도, 초보 모드에서는 기본적으로 각 명령 뒤에 짧은 해설 블록을 붙인다.[^5]
- 에러 메시지를 단순 번역이 아니라, 맥락과 함께 풀어서 설명하고, 관련 공식 문서나 이슈 링크를 요약해준다.[^1]
- 특정 작업(예: "인텔 맥에 Node 개발 환경 세팅")을 시나리오로 인식하면, 단계별 체크리스트를 제안하고 진행 상황을 추적한다.[^9]

### 5.4 세션 가이드 역할

- 한 세션 안에서의 과거 명령과 결과를 기억하고, "방금 전에 실행한 명령 기준으로 다음에 뭘 해야 할지"를 제안한다.[^3]
- 세션 요약 기능을 제공하여, 나중에 복습할 수 있도록 "오늘 한 작업 요약 + 주요 명령 리스트"를 출력하거나 파일로 저장한다.[^5]
- 필요 시 작업 전체를 공유할 수 있는 링크나 로그를 생성하여, 동료나 다른 AI에게 컨텍스트를 전달할 수 있게 한다.[^5]


## 6. 기능 정의

### 6.1 핵심 기능

1. **자연어 → 명령어 생성**  
   - 명령: `gai cmd "인텔 맥에서 Node LTS 설치하는 명령어 알려줘"`  
   - 동작: 
     - 사용자의 OS/아키텍처를 감지하거나 질의하여 적절한 설치 경로(nvm, asdf, 패키지 매니저 등)를 제안.[^7]
     - 각각의 후보 명령에 대해 용도, 장단점을 간략히 설명.  
     - 사용자가 선택한 옵션만 실제 실행 큐에 올림.

2. **명령 실행 + 결과 분석**  
   - 명령: `gai run` (직전에 제안된 명령 세트 실행) 또는 `gai explain-last` (직접 입력한 마지막 셸 명령 분석)  
   - 동작:  
     - 셸에서 명령 실행 후 출력과 종료 코드를 가져온다.[^3]
     - 성공 시: 무엇이 성공했는지, 다음에 보통 어떤 명령을 이어서 치는지 제안.  
     - 실패 시: 에러 메시지 요지, 흔한 원인, 추천 해결 절차를 단계별로 안내.

3. **에러 자동 감지 및 인터럽트 제안**  
   - 셸 프롬프트가 돌아올 때 마지막 명령의 종료 코드가 0이 아니면, 짧게 "에러 감지됨, 해석해 줄까? (y/n)"를 띄우는 옵션을 제공.[^6]
   - 사용자가 `y`를 선택하면, 해당 로그를 Gemini에 보내 해설과 수정 제안을 받음.

4. **단계별 가이드(워크플로 모드)**  
   - 명령: `gai guide "Node 개발 환경 처음부터 세팅"`  
   - 동작:  
     - 필요한 단계를 계획: "패키지 매니저 확인 → Node 설치 방법 선택 → 설치 → 버전 확인 → npm/yarn 확인" 등.[^9]
     - 각 단계마다 필요한 명령 제안, 실행, 검증, 해설을 한 싸이클로 제공.  
     - 중간에 실패할 경우, 해당 단계에서 재시도/우회 전략을 제안.

### 6.2 학습/설명 기능

- **명령어 해설 자동 출력 옵션**: 초보 모드에서는 각 명령 아래에 3~5줄 정도의 친절한 설명(무엇을, 왜, 언제 쓰는지)을 추가.[^5]
- **레벨별 설명**: 
  - "초보" 레벨: 비유와 일상 언어 포함, 개념 설명 위주.  
  - "중급" 레벨: 옵션, 플래그, 관련 도구 비교 중심.  
  - "고급" 레벨: 최소 설명, 관련 공식 문서 링크 요약만 제공.[^6]
- **세션 요약/노트 기능**: `gai summarize`로 오늘 세션의 핵심 명령과 이슈를 요약해 마크다운 파일로 저장, 추후 노션/위키에 올리기 쉽게 함.[^5]

### 6.3 안전/승인 UX

- **수동/자동 모드**  
  - Gemini CLI가 제공하는 `/mode`와 유사하게, "도구 사용 모드"를 자동/수동으로 전환할 수 있는 명령 제공.[^4][^5]
  - 기본값은 수동 모드로 두되, 사용자가 신뢰하는 범위(예: 읽기 전용 명령, `ls`, `cat`, `grep` 등)에 대해서는 자동 실행 화이트리스트를 둘 수 있음.

- **위험 명령 사전 경고**  
  - 파일 삭제, 시스템 설정 변경, 패키지 제거 등은 항상 명시적 확인을 요구.  
  - 단순 "accept" 대신, "이 명령은 ~/projects 폴더 아래 *.log 파일을 모두 삭제합니다. 진행할까요?" 같은 자연어 설명을 먼저 보여줌.[^7]

- **로그 및 롤백 힌트**  
  - 주요 변경 작업 전에는 가능한 경우 백업/스냅샷 명령을 함께 제안.  
  - 실패 시 "이전 상태로 되돌리는" 명령(예: Git 작업인 경우 `git restore`, `git reset` 등)을 함께 안내.[^2]


## 7. UX/인터랙션 설계 개요

### 7.1 인터페이스 형태

- **대화형 세션 모드**: `gai`를 실행하면 별도의 프롬프트가 열리고, 여기서 자연어로 질의/명령을 주고받는다.[^3]
- **원샷 명령 모드**: `gai cmd "…"`, `gai explain-last`처럼 단일 명령으로 요청/응답을 받고 종료하는 방식.[^6]
- **셸 훅 연동**: Zsh/Bash의 `precmd`/`preexec` 훅과 연동하여, 마지막 명령과 종료 코드를 자동으로 기록하고, 필요 시만 개입하는 형태.

### 7.2 피드백 스타일

- 텍스트는 기본적으로 한국어, 필요 시 원문 에러 메시지/영문 문서는 그대로 보여주고 위에 해설을 붙인다.[^5]
- 각 단계마다 "지금 잘 되고 있는지"를 명확히 표현하기 위해, 간단한 상태 아이콘(성공/주의/실패) 또는 태그를 텍스트로 표시한다.  
- 사용자가 원하면, 같은 상황을 다시 만났을 때 참고할 수 있도록 특정 단계를 "즐겨찾기"나 "북마크"로 저장하는 기능을 고려할 수 있다.[^9]


## 8. 기술/구현 방향(고레벨)

- **기반 엔진**:  
  - 1안: Google이 제공하는 공식 Gemini CLI를 그대로 설치하고, 그 위에 추가 명령/스크립트를 얹어 "튜터 모드"를 구현.[^4][^3]
  - 2안: Gemini API를 직접 사용하는 경량 CLI를 별도로 작성하되, ShellAI/AI Shell처럼 자연어→명령어 변환 패턴과 안전 확인 UX를 벤치마킹.[^7][^6]

- **환경 호환성**:  
  - 인텔 맥, Apple Silicon, 리눅스 등에서 모두 동작하도록, 아키텍처 의존성이 적은 언어(예: Python, Go, Node.js LTS)와 설치 경로를 선택.[^7]
  - 설치 스크립트에서 아키텍처와 OS를 감지하고, 가능한 경우 자동으로 최적의 설치 방법을 제안.

- **상태 관리**:  
  - 세션별 히스토리와 요약은 로컬 파일(예: `~/.gai/sessions/…`)에 저장.  
  - 민감한 정보는 저장하지 않도록, 마스킹/필터링 규칙을 두거나 사용자가 저장 여부를 선택할 수 있게 함.[^8]


## 9. 기대 효과

- 초보 개발자도 "복사·붙여넣기"를 넘어서, 명령의 의미와 시스템 동작을 이해하며 실험할 수 있다.[^5]
- 터미널만 열어두고도 질문, 명령, 학습, 트러블슈팅이 한 번에 가능해져, 브라우저/IDE 전환 비용이 줄어든다.[^1]
- 반복 에러(특히 환경 의존적인 문제)에 대해 원인 추적과 다음 액션 제안이 자동화되어, 시행착오 시간을 크게 줄일 수 있다.[^2]
- 공식 Gemini CLI 및 기존 오픈소스 도구(ShellAI, AI Shell, AskCLI 등)를 재활용함으로써, 구현 비용을 낮추면서도 원하는 "튜터형 CLI 에이전트" UX를 비교적 빠르게 실현할 수 있다.[^8][^6][^3][^7]
